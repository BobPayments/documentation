# Indo para Produção

Quando tudo estiver funcionando no sandbox, siga estes passos para ativar o ambiente de produção.

## Checklist pré-produção

- [ ] Secret Key armazenada em variável de ambiente (nunca hardcoded)
- [ ] Chave `sk_live_*` associada à integração correta (gateways de produção configurados)
- [ ] Modo dev desativado no dashboard
- [ ] Webhook configurado com URL HTTPS
- [ ] Assinatura de webhook sendo validada (HMAC-SHA256)
- [ ] Tratamento de webhooks duplicados (idempotência)
- [ ] Tratamento de erros HTTP 429 (rate limit) com retry
- [ ] Exibição do PIX como QR Code e copia-e-cola
- [ ] Testado fluxo completo no sandbox (criar, pagar, webhook)

## Passos para ativar produção

<Steps>
  <Step title="Crie uma chave de produção">
    No dashboard → seu projeto → **Configurações → Chaves de API → Nova chave**. Selecione o ambiente **live** e associe a chave à integração com os gateways de produção configurados.
  </Step>
  <Step title="Desative o modo sandbox">
    No dashboard → seu projeto → Configurações, desative o **modo sandbox**. Isso ativa as validações de produção.
  </Step>
  <Step title="Troque as chaves">
    Substitua `sk_test_*` por `sk_live_*` no seu backend. Nunca use chaves de produção no frontend.
  </Step>
  <Step title="Valide com um pagamento real">
    Teste com um pagamento de valor baixo (ex: R$ 1,00) para confirmar que todo o fluxo funciona: criação, webhook recebido e lógica de negócio executada.
  </Step>
</Steps>

## Idempotência e Deduplicação

A API possui mecanismos automáticos para prevenir a criação de transações duplicadas, protegendo contra cliques repetidos e falhas de rede.

### Deduplicação de transações

Quando uma transação PIX é criada, a API gera uma chave de deduplicação baseada em:

- **Projeto** (projectId)
- **Cliente** (documento CPF/CNPJ)
- **Valor** (amountCents)
- **Produto** (product)

Se uma transação com a mesma combinação já existir com status `waiting_payment`, a transação existente é retornada em vez de criar uma nova.

| Cenário | Resultado | HTTP |
|---------|-----------|------|
| Transação pendente com mesma combinação | Retorna a transação existente | `200` |
| Nenhuma transação pendente | Cria nova transação | `201` |
| Transação em processamento com mesma combinação | Erro de duplicata | `409` |

O cache de deduplicação é limpo automaticamente quando a transação atinge um status final (`paid`, `expired`, `cancelled`, `refunded`, `failed`).

Se uma transação idêntica estiver sendo processada simultaneamente, a API retorna HTTP `409`:

```json
{
  "type": "ERR_TRANSACTION_010",
  "title": "Transação Duplicada",
  "status": 409,
  "detail": "Já existe uma transação em processamento para este cliente e valor.",
  "instance": "/api/transactions"
}
```

<Note>
  Este erro é temporário. Aguarde alguns segundos e consulte o status da transação original com `GET /api/transactions/:id`.
</Note>

### Idempotência de webhooks

Cada notificação de webhook pode ser entregue mais de uma vez em casos raros. Use o `id` do recurso para garantir que o mesmo evento não seja processado duas vezes:

```javascript
app.post('/webhook', (req, res) => {
  const { event, data } = req.body;

  // Use o ID do recurso como chave de idempotência
  const alreadyProcessed = await checkIfProcessed(data.id, event);
  if (alreadyProcessed) {
    return res.status(200).json({ received: true });
  }

  await processEvent(event, data);
  await markAsProcessed(data.id, event);

  res.status(200).json({ received: true });
});
```

<AccordionGroup>
  <Accordion title="Sempre trate HTTP 409">
    Não crie nova transação ao receber 409. Consulte o status da transação existente.
  </Accordion>
  <Accordion title="Confie na deduplicação">
    É seguro reenviar a mesma requisição de criação de transação. A API retornará a transação existente se uma pendente for encontrada.
  </Accordion>
  <Accordion title="Processe webhooks de forma idempotente">
    Sempre verifique se o evento já foi processado antes de executar ações como liberar acesso ou enviar emails.
  </Accordion>
</AccordionGroup>

## Rate Limiting

A API aplica limites de requisição por cliente (documento CPF/CNPJ) para prevenir abuso e garantir estabilidade. O rate limit é aplicado no endpoint de criação de transações (`POST /api/transactions`) e é configurável por projeto via dashboard.

### Headers de resposta

| Header | Tipo | Descrição |
|--------|------|-----------|
| `X-RateLimit-Limit` | `integer` | Número máximo de requisições permitidas na janela |
| `X-RateLimit-Remaining` | `integer` | Requisições restantes na janela atual |
| `X-RateLimit-Reset` | `integer` | Timestamp Unix (segundos) de quando a janela reseta |
| `Retry-After` | `integer` | Segundos para aguardar (presente apenas em respostas 429) |

Se o limite for ultrapassado, a API retorna HTTP `429`:

```json
{
  "type": "ERR_RATE_LIMIT_001",
  "title": "Rate Limit Excedido",
  "status": 429,
  "detail": "Muitas requisições. Tente novamente em 30 segundos.",
  "instance": "/api/transactions"
}
```

### Tratando rate limit

<CodeGroup>

```javascript Node.js
async function createTransaction(data, retries = 3) {
  const response = await fetch('https://api.bobpayments.com/api/transactions', {
    method: 'POST',
    headers: {
      'Authorization': 'Bearer sk_live_xxx',
      'Content-Type': 'application/json',
    },
    body: JSON.stringify(data),
  });

  if (response.status === 429 && retries > 0) {
    const retryAfter = parseInt(response.headers.get('Retry-After') || '30');
    await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
    return createTransaction(data, retries - 1);
  }

  return response.json();
}
```

```python Python
import requests
import time

def create_transaction(data, retries=3):
    response = requests.post(
        'https://api.bobpayments.com/api/transactions',
        headers={'Authorization': 'Bearer sk_live_xxx'},
        json=data,
    )

    if response.status_code == 429 and retries > 0:
        retry_after = int(response.headers.get('Retry-After', 30))
        time.sleep(retry_after)
        return create_transaction(data, retries - 1)

    return response.json()
```

</CodeGroup>

<AccordionGroup>
  <Accordion title="Monitore os headers">
    Acompanhe `X-RateLimit-Remaining` para evitar atingir o limite. Implemente backoff proativo quando restar menos de 10% das requisições.
  </Accordion>
  <Accordion title="Use backoff exponencial">
    Em caso de 429, aguarde o tempo indicado em `Retry-After`. Se precisar implementar retry genérico, use backoff exponencial (1s, 2s, 4s, 8s...).
  </Accordion>
  <Accordion title="Não faça polling agressivo">
    Para verificar status de transações, use webhooks em vez de polling. Webhooks notificam em tempo real sem consumir sua cota de rate limit.
  </Accordion>
</AccordionGroup>

## Referência de Erros

Todas as respostas de erro seguem o formato [RFC 7807](https://datatracker.ietf.org/doc/html/rfc7807) (`application/problem+json`).

```json
{
  "type": "ERR_VALIDATION_001",
  "title": "Erro de Validação",
  "status": 400,
  "detail": "Erro de validação nos dados de body.",
  "instance": "/api/transactions"
}
```

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `type` | `string` | Código único do erro |
| `title` | `string` | Título legível do erro |
| `status` | `integer` | HTTP status code |
| `detail` | `string` | Descrição detalhada |
| `instance` | `string` | Path da requisição que gerou o erro |

### Autenticação

| HTTP | Código | Descrição |
|------|--------|-----------|
| 401 | `ERR_AUTH_009` | Token inválido ou ausente |

### Validação

| HTTP | Código | Descrição |
|------|--------|-----------|
| 400 | `ERR_VALIDATION_001` | Body ou query parameters inválidos |

Erros de validação incluem detalhes dos campos:

```json
{
  "type": "ERR_VALIDATION_001",
  "title": "Erro de Validação",
  "status": 400,
  "detail": "Erro de validação nos dados de body.",
  "instance": "/api/customers",
  "details": {
    "context": "body",
    "issues": [
      { "path": "email", "message": "Invalid email" }
    ]
  }
}
```

### Transações

| HTTP | Código | Descrição |
|------|--------|-----------|
| 404 | `ERR_TRANSACTION_001` | Transação não encontrada |
| 400 | `ERR_TRANSACTION_002` | Status da transação não permite esta operação |
| 403 | `ERR_TRANSACTION_003` | Transação pertence a outro projeto |
| 409 | `ERR_TRANSACTION_010` | Transação duplicada em processamento |

### Rate Limit

| HTTP | Código | Descrição |
|------|--------|-----------|
| 429 | `ERR_RATE_LIMIT_001` | Limite de requisições excedido. Verifique o header `Retry-After` |

### Gateways

| HTTP | Código | Descrição |
|------|--------|-----------|
| 503 | `ERR_ALL_GATEWAYS_FAILED` | Todos os gateways de pagamento falharam. Tente novamente em instantes |

### Fraude

| HTTP | Código | Descrição |
|------|--------|-----------|
| 403 | `FRAUD_DETECTED` | Limite de velocidade anti-fraude excedido para este cliente |

### Projetos

| HTTP | Código | Descrição |
|------|--------|-----------|
| 404 | `ERR_PROJECT_001` | Projeto não encontrado |

### Sandbox

| HTTP | Código | Descrição |
|------|--------|-----------|
| 400 | `ERR_SANDBOX_002` | Transação não pertence ao ambiente sandbox |

### HTTP Status Codes

| Status | Significado |
|--------|-------------|
| 200 | Sucesso |
| 201 | Recurso criado |
| 400 | Requisição inválida |
| 401 | Não autenticado |
| 403 | Não autorizado |
| 404 | Recurso não encontrado |
| 429 | Rate limit excedido |
| 500 | Erro interno do servidor |

<Note>
  Respostas `429` incluem o header `Retry-After` indicando quantos segundos aguardar antes de tentar novamente.
</Note>
